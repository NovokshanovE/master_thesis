\section{Реализация шардированного ART-хранилища}
\label{sec:art-storage}

\subsection{Общая идея шардирования}

Одно из ограничений общей in-memory структуры заключается в конкуренции
за блокировки. Если все операции чтения и записи проходят через одно
дерево и одну блокировку, потоковая модель быстро сталкивается с
конкуренцией за общий ресурс. Поэтому в работе применяется
шардирование.

Вставка и удаление выполняются по ключу конкретного префикса. Этот ключ
хешируется функцией FNV-32a, а значение хеша по модулю числа шардов
определяет, в какой шард попадёт запись. Каждый шард содержит собственное
ART-дерево и собственный \texttt{sync.RWMutex}. Такой подход локализует
модификацию конкретного префикса: обновления разных префиксов, попавших
в разные шарды, не конкурируют напрямую.

Важно, что шардирование не означает автоматической локализации
\texttt{Lookup} по IP-адресу в одном шарде. Поскольку шард выбирается по
ключу префикса, а не по исходному IP-адресу, разные кандидатные префиксы
для одного адреса могут находиться в разных шардах. Например, короткий
префикс и более специфичный префикс, покрывающие один и тот же адрес, не
обязаны иметь одинаковый хеш. Поэтому поиск наиболее специфичного
префикса описан отдельно в подразделе о \texttt{Lookup}.

\subsection{Внутреннее представление ключа}

Для хранения в ART IP-префикс переводится в байтовое представление,
пригодное для лексикографической навигации по дереву. ART работает с
последовательностью байтов, а не с абстрактным объектом «префикс»;
поэтому кодирование ключа определяет корректность поиска, вставки,
удаления и обхода~\cite{leis2013art}.

В реализации используется фиксированное кодирование адреса с явной
длиной маски. Адресная часть нормализуется по длине префикса, после чего
маска входит в ключ как отдельный компонент. Это позволяет различать,
например, записи \texttt{/24} и \texttt{/32}, даже если они относятся к
одной области адресного пространства, и выполнять точные операции
\texttt{Upsert} и \texttt{Delete} над конкретной префиксной записью.

\subsection{\texttt{Upsert} и \texttt{Delete}}

Основное преимущество нового хранилища проявляется на операциях
обновления. При \texttt{Upsert} сервис строит нормализованный ключ
префикса, выбирает по нему шард и локально модифицирует ART-дерево этого
шарда. \texttt{Delete} обрабатывает withdraw-событие тем же способом:
строится ключ удаляемого префикса, по нему выбирается шард, затем запись
удаляется из соответствующего дерева.

В отличие от варианта на \texttt{ranger}, эти операции не требуют полной
перестройки таблицы. Их стоимость зависит от длины ключа, размера
выбранного шарда и локальной формы ART-дерева, а не от полного числа
префиксов во всей таблице.

\subsection{\texttt{Lookup} по IP-адресу}

Точечный \texttt{Lookup} реализует longest prefix match по текущему
состоянию хранилища. Поскольку шарды выбираются по ключам префиксов,
поиск по IP-адресу не может быть сведён к выбору одного шарда по хешу
самого IP. В текущей реализации сервис формирует последовательность
кандидатных префиксов разной длины маски: от наиболее специфичной длины к
менее специфичным. Для каждого кандидата строится тот же ключ, который
использовался бы при вставке соответствующего префикса; затем по этому
ключу определяется шард и выполняется точный поиск в ART-дереве шарда.
Первый найденный кандидат является наиболее специфичным совпадающим
префиксом.

Для IPv4 проверяется не более 33 длин масок, для IPv6 -- не более 129.
Именно этим объясняется более высокая стоимость \texttt{Lookup} для IPv6
в экспериментальных результатах главы~\ref{sec:experiment-results}. При
этом \texttt{Upsert} и \texttt{Delete} остаются локальными для одного
шарда, а \texttt{Lookup} в общем случае может обращаться к нескольким
шардам.

\subsection{\texttt{Dump} полного состояния}

Полная выгрузка в ART-режиме реализуется как обход всех шардов и сборка
результатов. Это делает \texttt{Dump} дороже, чем в структуре, которая
держит уже готовый линейный массив записей. ART-подход платит за
изменяемость дополнительными затратами на обход, аллокации и
опциональную сортировку. В реализации сортировка управляется параметром
\texttt{ASMAP\_STORE\_ART\_DUMP\_SORT}; если порядок записей не важен,
отключение сортировки уменьшает время полной выгрузки на больших
таблицах.

Эта особенность не является ошибкой реализации. Она отражает компромисс:
структура, оптимизированная под частые инкрементальные обновления, редко
оказывается одновременно лучшей и для операции «отдать всё содержимое».

\subsection{Целевая операция и интерпретация результатов}

Оценивать новое хранилище только по времени \texttt{Dump} или по времени
точечного \texttt{Lookup} недостаточно. Целевая функция работы -- сделать
потоковую актуализацию практически применимой. Поэтому главными
измеряемыми параметрами являются CPU-время \texttt{Upsert} и
\texttt{Delete}; \texttt{Dump} и \texttt{Lookup} фиксируют стоимость,
которую приходится заплатить за изменяемость структуры. Эта установка
используется при интерпретации результатов в
главе~\ref{sec:experiment-results}.

\subsection{Согласованность параллельного доступа}

Шардирование само по себе не определяет модель согласованности; оно лишь
снижает конкуренцию за блокировки между независимыми операциями. Чтобы
\texttt{Lookup} и \texttt{Upsert} безопасно сосуществовали, в каждом
шарде используется \texttt{sync.RWMutex}. Все операции записи
(\texttt{Upsert}, \texttt{Delete}, локальная фаза \texttt{Replace})
удерживают блокировку на запись соответствующего шарда. \texttt{Lookup}
при проверке очередного кандидатного префикса удерживает блокировку на
чтение того шарда, куда попадает этот кандидат.

Такая дисциплина даёт следующие гарантии:

\begin{itemize}
\item внутри одного шарда параллельные операции чтения выполняются без
взаимного блокирования;
\item модификация одного шарда не блокирует операции в других шардах;
\item \texttt{Lookup} может последовательно обращаться к нескольким
шардам, но каждое отдельное обращение согласовано с локальными
изменениями соответствующего шарда;
\item \texttt{Dump} реализуется обходом всех шардов с локальными
блокировками на чтение.
\end{itemize}

Согласованность \texttt{Dump} в этой модели является шардовой, а не
глобальной атомарной согласованностью всей таблицы: каждое значение,
попавшее в выгрузку, корректно относительно своего шарда в момент чтения.
Для прикладного сценария получения полной таблицы это рассматривается как
текущее представление сервиса. Если клиенту требуется строгое
восстановление состояния во времени, он должен использовать сценарий
«полный снимок + последующая подписка на изменения», описанный в
главе~\ref{sec:snapshot-stream}.

\subsection{Поведение во время ресинхронизации}

Отдельный режим работы -- ресинхронизация. В этот момент сервис должен:

\begin{enumerate}
\item получить новый снимок;
\item подготовить из него корректное внутреннее представление;
\item заменить текущее состояние хранилища на новое;
\item возобновить обработку потока обновлений.
\end{enumerate}

В шардированной реализации эта последовательность выполняется как
скоординированная замена содержимого всех шардов. Параллельный
\texttt{Lookup}, попадающий в момент перестроения, либо обслуживается из
состояния соответствующего шарда до замены, либо ожидает завершения
локальной фазы записи. Кратковременные различия видимости между шардами в
момент ресинхронизации являются следствием выбранной модели и допустимы,
поскольку ресинхронизация происходит только после сбоев или длительных
пауз, а затем сервис снова переходит к штатному потоку обновлений.
